# Redis — 개념 정리 및 이 프로젝트에서의 쓰임

Redis 자체에 대한 개념 정리 + 이 프로젝트에서 현재/예정된 사용처를 함께 정리한 문서.
코드 위치는 전부 `server/src/main/java/com/example/demo/` 기준.

## Redis란

메모리(RAM)에 데이터를 올려두고 쓰는 key-value 저장소. Postgres처럼 별도 서버
프로세스로 떠 있고, 앱은 클라이언트로 접속해서 명령을 주고받는다.

- **빠른 이유**: 디스크가 아니라 메모리 접근이고, 명령 처리가 싱글 스레드라
  락 경합이 없다.
- **트레이드오프**: 메모리 용량만큼만 담을 수 있고, 기본적으로 서버가 재시작되면
  데이터가 날아갈 수 있다(디스크 스냅샷/로그로 영속화하는 옵션도 있지만, 이
  프로젝트는 그 옵션을 쓰지 않는다). 그래서 "없어져도 다시 만들 수 있는 데이터"나
  "잠깐만 유효하면 되는 데이터"에 적합하고, 진짜 원본 데이터(사용자 정보, 일정
  등)는 항상 Postgres에 둔다.

## 자료구조 종류

Key-Value인데 값의 자료구조를 여러 종류 중에 골라 쓸 수 있다:

| 타입 | 생김새 | 쓰임 예 |
|---|---|---|
| String | `key → "문자열 하나"` | 가장 단순한 캐싱 |
| List | `key → [값1, 값2, ...]` (순서 있음) | 최근 항목 목록 |
| Hash | `key → {필드1: 값1, 필드2: 값2}` | 미니 객체 하나 통째로 캐싱 |
| Set | `key → {값1, 값2, ...}` (중복 없음) | "본 적 있는 id 집합" 같은 것 |
| Sorted Set (ZSet) | `key → {값: 점수, ...}` (점수순 자동 정렬) | 랭킹/집계 |

## 언제 필요한가 — 판단 기준

다음 세 가지를 동시에 만족하면 Redis 캐싱을 고려한다:
1. 자주 조회된다
2. 다시 계산/조회하는 비용이 크다 (외부 API 호출, 무거운 쿼리 등)
3. 몇 분~몇 시간 지난 데이터라도 괜찮다 (실시간으로 항상 최신일 필요는 없음)

## 캐싱 동작 원리 — cache-aside 패턴

"먼저 캐시에 있나 찾아보고, 없으면 원본을 호출해서 그 결과를 캐시에 넣어두는"
방식을 **cache-aside**라고 부른다. 핵심은 **같은 입력이면 항상 같은 키가
나와야 한다**는 것 — 그래야 나중에 그 키로 다시 찾을 수 있다.

- 저장할 때: `대화텍스트A` → 해시 → `embedding:9f86d081...` → 이 키에 벡터값 저장
- 나중에 똑같은 `대화텍스트A`가 다시 들어오면 → 똑같은 해시 함수를 거치므로 →
  똑같이 `embedding:9f86d081...`가 나옴 → 그 키로 조회하면 아까 저장해둔 값이
  바로 나옴 (원본 API 호출 없이)
- 대화텍스트가 조금이라도 다르면(`대화텍스트B`) → 다른 해시(`embedding:3a7c...`)
  → 캐시에 없음 → 그때만 진짜 원본(Gemini) 호출

## 캐싱 방식 두 가지 — 완전일치 vs 유사도

| | 캐시 대상 | 캐싱 방식 |
|---|---|---|
| 일반적인 웹 캐싱 (대부분) | DB 조회 결과, API 응답, 세션 등 | **완전일치** — 입력을 그대로(또는 해시해서) 키로 씀 |
| LLM 답변 캐싱 (특수 케이스) | 챗봇의 최종 텍스트 답변 | **유사도 캐싱** — 별도의 저비용 임베딩으로 "비슷한 질문"을 찾아 재사용 |

**일반적인 캐싱(DB 쿼리 결과, API 응답, 세션 등)은 압도적으로 완전일치
방식이다.** `key = 입력을 해시한 값`이고, 입력이 한 글자라도 다르면 다른
키가 나온다. 이 문서의 cache-aside 패턴, 그리고 이 프로젝트의 임베딩
캐싱도 이 방식이다.

**유사도(semantic) 캐싱**은 GPTCache, Redis의 semantic cache 기능 같은
도구로 실제로 쓰이는 기법이지만, 보통 **임베딩을 만드는 단계가 아니라
비싼 LLM 답변(채팅 완성) 자체를 캐싱할 때** 쓴다 — 예: 고객센터 챗봇이
"배송 언제 와요?"와 "배송 얼마나 걸려요?"를 비슷한 질문으로 인식해서
이미 만든 답변을 재사용. 여기선 "질문을 임베딩해서 유사도 비교"하는
비용은 작고, 그 대신 진짜 비싼 건 큰 모델이 답변을 새로 생성하는
것이라 "싼 임베딩 비교로 비싼 생성 호출을 건너뛴다"는 계산이 맞아떨어진다.

반면 이 프로젝트에서 캐싱하려는 대상은 **임베딩 벡터 그 자체**다. "임베딩을
만들지 말지 결정하려고 임베딩을 또 만드는" 순환이 되므로 유사도 캐싱을
그대로 가져다 쓸 수 없다 — 아래 "한계" 항목에서 자세히 다룬다.

## 이 프로젝트에서의 구성

- **배포**: `docker run redis:7-alpine`, 포트 `6379` (로컬 개발은 `/dev-up`
  스킬이 컨테이너를 기동). 접속 정보는 `application.properties`의
  `spring.data.redis.host`/`spring.data.redis.port`.
- **의존성**: `spring-boot-starter-data-redis` (`build.gradle`).
- **코드**: `redis/RedisService.java` — `StringRedisTemplate` 기반으로 String
  타입만 다루는 단순 `save(key, value)` / `get(key)`, TTL 1시간 고정.

### 현재 구현

**Gemini 임베딩 캐싱** (`ai/AiService.java`)

`createEmbedding(text)`가 cache-aside로 동작한다. 벡터를 이미 문자열 그대로
반환하므로(`response.embedding.values.toString()` → `"[0.0123, -0.045, ...]"`)
별도 직렬화 없이 그 문자열을 그대로 캐싱한다.

- **캐시 키**: `conversationText`를 그대로 키로 쓰지 않고, `cacheKey(text)`가
  SHA-256으로 해시해서 `embedding:<해시값>` 형태로 만든다. 대화 텍스트는 길고
  매번 달라지는데, 해시는 입력이 같으면 항상 같은 짧고 고정된 길이의 문자열이
  나오기 때문에 Redis 키로 다루기 편하다.
- **캐시 값**: `createEmbedding`이 반환하는 문자열(벡터) 그대로.
- **흐름**:
  ```java
  public String createEmbedding(String text) {
      String key = cacheKey(text);
      String cached = redisService.get(key);
      if (cached != null) return cached;   // 캐시 히트 — Gemini 호출 생략

      // 캐시 미스 — 기존 로직대로 Gemini 호출
      String result = /* RestClient 호출 결과 */;
      redisService.save(key, result);      // 다음 호출을 위해 저장
      return result;
  }
  ```
- **TTL**: `RedisService.save`의 기존 1시간 TTL을 그대로 재사용.
- **로그**: 캐시 히트/미스를 `[embedding cache] hit/miss: <key>`로 남긴다.

**한계 — 완전 일치만 캐시된다 (의도된 설계)**

캐시 키는 `conversationText`(대화 히스토리 + 새 메시지를 이어붙인 전체 문자열)를
SHA-256으로 해시한 값이다. 그래서 "카페 가고싶어"와 "카페 갈래"처럼 사람이
보기엔 같은 의도라도 글자가 다르면 완전히 다른 키가 나와 캐시가 안 맞는다 —
의미 기반 유사도 캐싱이 아니라 단순 완전 일치 캐싱이다. 이 프로젝트는 매 요청마다
`conversationText`가 대화 진행에 따라 계속 길어지고 매번 달라지므로, 실제로
캐시가 맞아떨어지는 경우는 주로 완전히 중복된 요청(네트워크 재시도, 중복 클릭)
이거나 서로 다른 세션의 짧은 초반 대화가 우연히 글자 그대로 겹치는 경우 정도다.

의미 기반 유사도 캐싱(위 섹션 참고)은 의도적으로 채택하지 않았다:
- **순환 문제**: 유사도를 판단하려면 이 텍스트도 먼저 벡터화해야 하는데, 그게
  바로 캐싱으로 피하려던 API 호출이다. (로컬에 별도의 경량 임베딩 모델을 따로
  띄우면 피할 수 있지만, 이 프로젝트 규모에 비해 인프라가 과함.)
- **정확성 위험**: "카페 갈래"와 "카페 가지마"처럼 글자는 비슷해도 의미가
  반대인 경우, 유사도 캐싱은 조용히 틀린 벡터를 재사용할 수 있다. 완전 일치
  캐싱은 미스가 나면 그냥 다시 계산할 뿐이라 느릴지언정 틀리지는 않는다.
- **이미 다른 계층이 의미를 처리함**: 캐싱된 벡터는 결국
  `JejuPlaceRepository.findSimilarPlacesWithFilter`의 pgvector 코사인 거리
  검색에 쓰인다(`docs/PLACE_SEARCH.md`). "의미가 비슷한 걸 찾는" 역할은 이미
  이 검색 단계가 맡고 있어서, 캐싱 레이어까지 의미를 이해할 필요가 없다.

### 예정된 사용처

**인기 장소 랭킹** (백엔드 로드맵, 미구현)

조회/위시리스트/일정 포함 이벤트로 점수를 누적하는 Sorted Set. `RedisService`에
`ZSetOperations` 기반 메서드가 추가될 예정.

**장소 검색 결과 캐싱** (미구현, 범위 확정 안 됨)

`JejuPlaceRepository.findSimilarPlacesWithFilter` 결과도 캐싱을 고려할 수
있으나, region/category 조합만큼 키 공간이 커지고 `jeju_place` 데이터가
갱신될 때(TourAPI 재적재, 임베딩 백필) stale 캐시 문제가 더 복잡해서 아직
손대지 않았다.
