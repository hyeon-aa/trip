# PRD — 제주 여행 플래너

기술 스택/아키텍처는 `CLAUDE.md`(영문, Claude Code용)와 `docs/PROJECT.md`를 참고.
이 문서는 "무엇을, 왜" 만들었는지를 기능 단위로 정리한 한국어 문서다. 기능이
바뀌면 이 문서도 같이 업데이트한다.

## 개요

**제품**: AI 챗봇과 대화하며 제주도 여행 일정을 만드는 서비스.
**타겟 사용자**: 제주 여행을 계획 중이고, 검색으로 일일이 장소를 찾아 직접 동선을
짜기보다 대화로 빠르게 일정을 받고 싶은 사람.
**핵심 가치**: 실제 존재하는 제주 장소 데이터(TourAPI 기반)만 추천하고, 이동 동선까지
자동으로 최적화해서 보여준다 — AI가 지어낸 가짜 장소나 비현실적인 동선을 주지 않는다.

## 사용자 플로우

1. 메인 화면(`/`) 진입 — 지도, 일정 칼럼, 채팅 칼럼이 한 화면에 보인다.
2. (선택) 헤더의 "위시리스트" 링크로 `/wishlist`로 이동해 가고 싶은 장소를 미리
   검색해서 저장해둔다. 저장한 장소는 지도에 항상 빨간 핀으로 표시된다.
3. 메인 화면으로 돌아와 채팅으로 여행 정보를 알려준다 — 온보딩 단계(아래 "기능 3"
   참고)를 버튼/직접 입력으로 진행.
4. 마지막 온보딩 답변(숙소)을 넣는 순간 AI가 일정을 생성한다 — 지도에 일차별 마커·
   동선이, 일정 칼럼에 장소별 상세가 뜬다.
5. 일정 칼럼의 Day 탭으로 원하는 날짜만 골라서 볼 수 있다 — 지도도 그 날짜만 표시.
6. 채팅으로 "1일차에 카페 하나 더 넣어줘" 같은 수정 요청을 하면, 맥락을 유지한 채
   일정 전체가 다시 생성되어 갱신된다.

## 기능별 요구사항

### 1. 위시리스트 (`/wishlist`)
- **요구사항**: 카카오 장소 검색으로 원하는 장소를 찾아 저장/삭제할 수 있어야 한다.
- **왜 별도 페이지인가**: 검색/저장은 지도를 직접 볼 필요가 없는 독립적인 작업이다.
  메인 화면(채팅 입력 포함)과 분리해서, 채팅 영역이 위시리스트 UI에 눌려 좁아지지
  않게 한다.
- **저장된 장소의 역할**: 위시리스트에 저장한 장소는 AI가 일정을 짤 때 추천 후보에도
  같이 들어간다 (TourAPI 장소와 동일한 취급).

### 2. 지도 (`KakaoMap`, 메인 화면)
- **요구사항**: 위시리스트 장소, 생성된 일정의 장소/동선을 한 화면에서 봐야 한다.
  여러 날짜의 동선이 한꺼번에 겹쳐 보이지 않고, 선택한 날짜의 동선만 명확히
  구분되어야 한다.
- Day 탭으로 선택한 날짜의 마커·동선만 표시하고 다른 날짜는 숨긴다.
- 일차(Day)별로 다른 색상(`getDayColor`, 7색 팔레트, 색각이상 검증됨)을 써서 지도
  마커/동선과 일정 칼럼 표시를 시각적으로 맞춘다. **알려진 제한**: 팔레트가 7색
  까지만 구분되므로 8일 이상 여행은 순환되어 두 날짜가 같은 색을 공유할 수 있다.

### 3. 채팅 온보딩 (`PlanChat`)
- **요구사항**: 여행 스타일/동행자/기간/도착 시간/출발 시간/숙소, 이 6가지를 받아야
  AI가 제대로 된 일정을 만들 수 있다.
- **입력 방식**: 매번 문장으로 타이핑하지 않아도 되어야 한다 — 스타일은 다중 선택
  칩, 동행자/기간은 단일 선택 버튼, 도착/출발 시간은 프리셋 버튼 + 정확한 시:분
  선택용 시계 위젯, 숙소는 카카오 장소검색으로 고른다. 어느 단계든 직접 타이핑도
  항상 가능해야 한다.
- **UI 형태**: 채팅 칼럼의 같은 자리에서, 6단계 모두 진행률 표시(N/6)와 "이전"
  버튼이 있는 위자드 카드로 보여준다(버블이 계속 쌓이는 방식이 아님). "이전"으로
  돌아가면 그 단계에서 골랐던 답변이 그대로 복원되어 바로 바꿀 수 있다. 마지막
  단계(숙소) 완료 후에는 지금까지의 대화가 일반 채팅 버블 흐름으로 자연스럽게
  전환된다.
- **서버 호출 비용**: `/plan/chat` 호출 한 번마다 임베딩 생성 + Gemini 채팅 호출이
  들어가므로, 6단계 전부를 매번 서버에 묻지 않아야 한다. 스타일~출발시간(5단계)은
  프론트에서만 로컬로 진행하고, 마지막 숙소 단계에서만 지금까지의 대화 전체를
  히스토리로 얹어 실제 API를 정확히 한 번 호출한다.

### 4. 일정 생성 (AI, `/plan/chat`)
- **요구사항**: 사용자가 준 조건(스타일/동행자/기간/시간/숙소)에 맞춰, DB에 실제로
  있는 제주 장소만 골라 일자별 일정을 만들어야 한다. 존재하지 않는 장소나 좌표를
  지어내면 안 된다.
- **하루 장소 개수**: 최소 3곳, 최대 5곳(등산 등 장시간 활동이 있는 날은 3곳 위주).
  휴식 위주 여행이라도 일정이 지나치게 헐거워 보이지 않도록 최소 개수를 보장한다.
- **중복 방지**: 같은 장소가 서로 다른 날짜에 중복 배정되지 않아야 한다.
- **지역 쏠림 방지**: 하루 일정이 여러 지역에 흩어지지 않아야 한다 — 동부/서부/남부/
  제주시 4대 권역을 안 섞는 것은 물론, 그 안에서도 읍/면/동 단위(예: 애월읍, 성산읍)
  까지 최대한 통일해서 하루 동선이 좁은 지역에 모이게 한다.
- **도착/출발 시간 반영**: 첫날엔 도착 시간 이전에 일정을 넣지 않고, 마지막 날엔
  출발(비행기) 시간에 늦지 않게 공항 이동 여유를 확보한다. 일정을 만드는 1차 AI 호출과
  각 장소의 실제 방문 시간을 정하는 2차 AI 호출이 분리되어 있으므로, 두 호출 모두
  같은 시간 제약(대화 맥락 + 첫날/마지막날 여부)을 공유해야 한다 — 그래야 "늦은 시간
  도착"이라는 설명과 실제 배정 시간이 항상 일치한다.

### 5. 동선 최적화 (서버, `optimalOrder`)
- **요구사항**: 하루 안에서 방문 순서를 실제 이동 거리가 최소가 되도록 정렬해야 한다.
- **숙소 기준 왕복**: 숙소 위치를 입력했다면, 마지막 날을 제외한 모든 날은 숙소에서
  출발해 숙소로 돌아오는 왕복 개념으로 거리를 계산한다. 마지막 날만 종료 지점이
  숙소 대신 제주공항이 된다. 숙소를 입력하지 않은 경우에는 순수 방문지 간 거리만
  본다.

### 6. 일정 패널 (`SchedulePanel`)
- **요구사항**: 선택한 날짜의 방문 장소, 순서, 추천 방문 시간, 추천 이유를 확인할 수
  있어야 한다. Day 탭으로 날짜를 바꿀 수 있다 (지도와 연동, "기능 2" 참고).

### 7. 장소 데이터 운영 (관리용, `/jeju/*`)
- **요구사항**: AI가 추천할 실제 장소 데이터가 최신이고, 검색에 필요한 임베딩/지역
  정보가 빠짐없이 채워져 있어야 한다.
- TourAPI에서 장소 수집(`/jeju/init/all`) → 임베딩 생성(`/jeju/embedding`, 100개씩
  반복 호출 필요) → 읍/면/동 세분화(`/jeju/sub-region`, 카카오 역지오코딩, 한 번
  호출로 전부 처리) 순서로 운영한다.
- **데이터 정합성**: TourAPI가 숙박 시설(리조트/펜션 등)을 "관광지"로 잘못 분류해
  내려주는 경우가 있으므로, 일정 추천 후보에서 숙박 시설은 걸러내야 한다. 좌표가
  대한민국 범위를 벗어나는 등 명백히 잘못된 장소는 주소 기준으로 재지오코딩해
  바로잡아야 한다.

## 비기능 요구사항 / 제약

- **인증 없음**: 로그인 기능이 없다 — 의도된 범위 밖 (`SecurityConfig`가 모든 요청
  허용). 위시리스트도 사용자 구분 없이 전역 테이블 하나를 공유한다(1인 사용 전제).
- **배포 도메인**: 현재 CORS가 `localhost:3000`만 허용 — 배포하려면 이 설정부터
  바꿔야 한다.

## 알려진 제한사항 / 향후 과제

- Day 팔레트가 7색까지만 구분되고 8일 이상 여행은 색이 겹친다 (기능 2 참고).
- `region`(4대 권역)이 위경도 임계값 근사치라 경계 근처 장소가 실제 행정구역과 다르게
  분류될 수 있다 — `sub_region`(읍/면/동)이 더 정확하니 가능하면 그쪽을 신뢰할 것.
- 숙박(호텔/리조트/펜션) 데이터는 TourAPI에서 아예 수집하지 않는다 — 위 "기능 7"에서
  다룬 미스매치 정리는 사후 필터링일 뿐, 근본적으로 숙박용 장소 카테고리를 새로
  수집하는 기능은 없음.
