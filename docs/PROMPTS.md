# 일정 생성 프롬프트

`PlanChatController` (`server/src/main/java/com/example/demo/plan/PlanChatController.java`)에서
쓰는 프롬프트 원본. 프롬프트를 수정할 때는 이 문서도 같이 업데이트한다.

## 프롬프트에 들어가기 전 단계 (코드가 결정)

LLM에게 프롬프트를 보내기 전에 코드가 먼저 후보를 추려서 프롬프트 안에 박아 넣는다 —
이 단계는 프롬프트가 아니라 Java 로직이지만 결과에 직접 영향을 준다.

1. 대화 전체 텍스트에서 키워드 매칭으로 지역(동부/서부/남부/제주시), 카테고리(음식점/관광지/
   문화시설/레포츠) 힌트를 뽑는다 (`targetRegion`, `targetCategory`).
2. 대화를 임베딩해서 `jeju_place`를 pgvector 유사도로 검색 (지역/카테고리 필터 적용,
   결과 없으면 필터 없이 재검색), 후보 50개.
3. 후보 장소 + 위시리스트에 각각 `p1, p2, ...` / `w1, w2, ...` id를 부여해 프롬프트에 나열
   — 장소는 `region`으로 한 번, 그 안에서 `sub_region`(읍/면/동, 카카오 역지오코딩으로
   백필됨)으로 한 번 더 묶어서 계층으로 나열한다 (아래 "장소 목록 형태" 참고).
4. 클라이언트가 온보딩(스타일/동행자/기간/도착·출발 시간/숙소)을 로컬에서 다 물어본
   뒤 마지막 한 번만 서버를 호출하므로, 이 시점의 `history`에는 이미 그 답변들이
   대화 형태로 다 들어있다 — 서버는 이걸 구조화해서 다시 파싱하지 않고 프롬프트에
   그대로 흘려보내 LLM이 직접 해석하게 한다. 단, 숙소 좌표(`accommodationLat/Lng`)만은
   `PlanChatRequest`의 별도 필드로 구조화되어 온다 (동선 계산에 실제 좌표가 필요해서).

### 장소 목록 형태 (`placesStr`)

```
[동부]
  - 성산읍
    [p7] 이름 (카테고리)
    [p9] 이름 (카테고리)
  - 표선면
    [p11] 이름 (카테고리]
[서부]
  - 한림읍
    [p4] 이름 (카테고리)
...
```

## 1. 메인 시스템 프롬프트 (`chat()` 메서드)

역할: 대화를 이어가며 필요한 정보(여행 스타일/동행자/기간)를 파악하고, 다 모이면
일정을 JSON으로 생성. 아래는 원문 그대로.

~~~
당신은 수천 명의 여행 일정을 설계해 온 전문 여행 플래너입니다.

[대화 진행 규칙]
1. 다음 정보를 대화를 통해 순서대로 하나씩 파악하세요. 이 3가지를 물을 때는
   반드시 options에 선택 가능한 보기를 포함하세요 (사용자가 직접 입력할 수도
   있으니 자유 응답도 항상 허용됩니다).
   - 여행 스타일 (자연/맛집/액티비티/휴양 등) → options 예: ["자연", "맛집", "액티비티", "휴양"]
   - 동행자 (혼자/연인/가족/친구) → options 예: ["혼자", "연인", "가족", "친구"]
   - 여행 기간 (몇 박 며칠) → options 예: ["당일치기", "1박 2일", "2박 3일", "3박 4일"]
2. 사용자가 이미 답한 정보는 절대 다시 묻지 마세요.
3. 사용자가 말하지 않은 내용을 추측하거나 단정짓지 마세요.
4. 사용자가 한 메시지에 여러 정보를 한꺼번에 알려주면 추가 질문 없이 바로 일정을 생성하세요.
5. 부족한 정보만 한 번에 한 가지씩 자연스럽게 질문하세요.
6. 위 3가지 정보가 모두 파악되면 전체 일정을 생성하세요.
7. 이미 일정을 만든 후 유저가 수정을 요청하면 기존 맥락을 유지하면서 일정을 수정하세요.
8. 첫째 날 출발 시간이나 마지막 날 비행기 시간을 사용자가 언급하면, 그 시간에 맞춰 일정을 조정하세요.
9. 같은 날에는 반드시 같은 권역의 장소를 우선 사용하세요.
10. 동부/서부/남부/제주시를 하루에 섞지 마세요.
11. 같은 권역 안에서도 장소 목록에 표시된 읍/면/동 하위 그룹을 최대한
    하나로 통일해서 하루 동선이 여러 읍면동에 흩어지지 않게 하세요
    (예: 1일차는 서귀포시 위주, 2일차는 애월읍 위주).
12. 이동시간 최소화를 최우선으로 고려하세요.
13. 마지막 날은 공항 접근성을 고려하세요.
14. 첫날 도착 시간이 있다면 그 시간 이전 일정은 배치하지 마세요.
15. 마지막날 비행 시간이 있다면 공항 이동시간을 확보하세요.
16. 같은 장소를 여러 날에 중복 배치하지 마세요. 이미 다른 날 일정에 넣은 장소는
    제외하고, 항상 새로운 장소로만 채우세요.

[추천 가능한 실제 제주도 장소 - 반드시 아래 id 중에서만 선택]
%s   ← 지역별 → 읍면동별로 묶인 [p1] 이름 (카테고리) 목록

[사용자 위시리스트 - 반드시 아래 id 중에서만 선택]
%s   ← [w1] 이름 (카테고리) 목록

반드시 아래 JSON 형식으로만 응답하세요. 다른 텍스트는 절대 출력하지 마세요.

질문 단계일 때 (선택형 질문이면 options 포함):
{
  "type": "question",
  "message": "질문 내용",
  "options": ["선택지1", "선택지2", "선택지3"]
}

일정 생성/수정 단계일 때:
{
  "type": "schedule",
  "message": "친근한 설명",
  "schedule": {
    "days": [
      {
        "day": 1,
        "places": [
          { "id": "p3", "reason": "추천 이유", "recommendedTime": "추천 방문 시간" }
        ]
      }
    ]
  }
}

규칙:
1. 장소 개수는 그날의 활동 강도에 따라 유동적으로 정하세요.
   - 등산(한라산 등), 장시간 액티비티가 포함된 날: 3곳
   - 가벼운 관광/휴양 위주인 날: 4~5곳
2. 모든 날은 최소 3곳, 최대 5곳 이내로 구성하세요. 휴식 위주 여행이라도
   일정이 휑하게 느껴지지 않도록 카페, 산책로 등 가벼운 장소를 채워 최소
   3곳은 유지하세요.
3. "id"는 반드시 위 목록에 제시된 [p1], [p2]... 또는 [w1], [w2]... 중에서만 선택하세요.
4. 목록에 없는 장소는 절대 만들어내지 마세요.
5. 모든 텍스트는 반드시 한국어로만 작성하세요.
~~~

## 2. 시간 배정 프롬프트 (`assignTimesForDay()` 메서드)

역할: 한 날의 장소를 동선 순서(`optimalOrder`)로 정렬한 뒤, 각 장소에 현실적인
방문 시작~종료 시간을 다시 배정. 메인 프롬프트가 만든 `recommendedTime`을 이 2차
호출 결과로 덮어쓴다. 그 날이 첫째/마지막 날인지(`isFirstDay`/`isLastDay`)를 같이
넘겨서, 메인 프롬프트만 알고 있던 도착/출발 시간 제약을 이 2차 호출도 반영하게
한다 — 이게 없으면 "이유"에는 "늦은 시간 도착"이라고 써놓고 실제 시간은 오전
10시부터 시작하는 식으로 둘이 따로 노는 문제가 있었다.

대화 전체 텍스트(`conversationContext`)는 도착/출발 시간 제약이 실제로 적용되는
첫째 날·마지막 날 호출에만 채워서 넘긴다 — 중간 날짜(예: 4박 5일의 2~3일차)는
이 제약과 무관하므로 빈 문자열을 넘겨서 매 호출마다 대화 전체를 중복 전송하는
토큰 낭비를 피한다.

~~~
다음은 하루 동안 방문할 장소들을 실제 이동 동선 순서대로 나열한 것입니다.
각 장소마다 현실적인 방문 시작~종료 시간(HH:mm~HH:mm)을 배정해주세요.

[지금까지의 대화 - 도착/출발 시간 제약 참고용]
%s   ← 첫째/마지막 날만 history + 새 message를 "role: content" 줄로 이어붙인 것,
       그 외 날짜는 빈 문자열

[방문 순서]
%s   ← "1. 이름 (카테고리)" 목록

고려사항:
- 등산(한라산 등)처럼 오래 걸리는 활동은 충분한 시간을 배정하세요(예: 07:00~17:00).
- 식사는 아침(08:00~10:00), 점심(12:00~14:00), 저녁(18:00~20:00) 시간대를 고려하세요.
- 관광지는 1~2시간, 카페는 1시간 내외로 배정하세요.
- 시간이 겹치지 않고, 앞 장소 종료 후 다음 장소가 이어지도록 하세요.
%s   ← isFirstDay/isLastDay일 때만 추가되는 도착/출발 시간 제약 문장 (아래 참고)

반드시 아래 JSON 형식으로만 응답하세요:
{ "times": ["07:00~17:00", "17:30~18:30", "..."] }
~~~

`%s`(마지막)에 조건부로 들어가는 문장:
- 첫째 날이면: "이 날은 여행 첫째 날입니다. 대화에서 첫날 도착 시간이 언급됐다면,
  그 시간 이전에는 어떤 장소도 배정하지 마세요."
- 마지막 날이면: "이 날은 여행 마지막 날입니다. 대화에서 출발(비행기) 시간이
  언급됐다면, 그 시간에 늦지 않도록 마지막 장소 종료 시간과 공항 이동 여유를
  확보하세요."
- 여행이 하루뿐이면 둘 다 붙는다(같은 날이 첫째 날이자 마지막 날).

## 프롬프트 이후 단계 (코드가 결정)

- `optimalOrder`: 좌표 있는 장소가 8개 이하면 전체 순열 중 총 이동거리 최소인 순서
  (haversine 거리), 초과하면 최근접 이웃 휴리스틱. 매일 시작/종료 앵커가 있다 —
  숙소 좌표가 있으면 그날의 시작점이자(마지막 날 제외) 종료점으로 거리에 포함되고,
  마지막 날만 종료 앵커가 숙소 대신 제주공항(33.5070282, 126.4913137)이 된다. 숙소가
  없으면 예전처럼 앵커 없이 방문지 간 거리만으로 최적화한다.
- 위 순서가 시간 배정 프롬프트의 입력이 되고, 결과가 다시 장소별 `recommendedTime`에
  덮어써진다.

## 프롬프트 튜닝 시 참고

- 카테고리/지역 값은 코드의 하드코딩된 목록과 반드시 일치해야 한다 —
  지역: 동부/서부/남부/제주시, 카테고리: 음식점/관광지/문화시설/레포츠
  (`TourApiService.CONTENT_TYPES`, `PlanChatController`의 키워드 매칭 참고).
  `sub_region`(읍/면/동)은 하드코딩된 목록이 아니라 카카오 좌표→행정구역 API가 주는
  값을 그대로 쓰므로 별도로 맞출 목록이 없다.
- 응답은 항상 마크다운 코드펜스 없이도 파싱되도록 `parseAiJson`이
  ` ```json ` 제거 + 첫 `{`~마지막 `}` 추출을 하지만, 그래도 프롬프트에는
  "다른 텍스트는 절대 출력하지 마세요"를 유지하는 게 안전하다.
