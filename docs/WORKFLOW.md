# 기능 개발 흐름

기능 하나를 백로그에 올리는 것부터 main에 머지하는 것까지, 이 프로젝트에서 실제로
쓰는 흐름을 정리한다. 각 단계는 `.claude/skills/`의 스킬로 뒷받침된다 — 새 스킬을
추가하거나 흐름이 바뀌면 이 문서도 같이 업데이트한다.

## 전체 흐름

```
1. 백로그(이슈)  →  2. 브랜치 생성  →  3. PRD에 요구사항 추가  →
4. 구현  →  5. PR 생성 + 자동 리뷰  →  6. 확인 후 머지  →  7. 브랜치 정리
```

| 단계 | 자동화 여부 | 담당 |
|---|---|---|
| 1. 백로그(이슈) | ✅ `/issue` | 초안 작성 + `gh issue create` |
| 2. 브랜치 생성 | ✅ `/issue` 4단계에 포함 | 이슈를 고르면 스킬이 브랜치까지 만듦 |
| 3. PRD 업데이트 | 안내만 자동 | 실제 내용은 대화로 같이 정함 |
| 4. 구현 | ❌ 수동 | 대화로 같이 코드 작성 |
| 5. PR + 리뷰 | ✅ `/pr` | PR 생성 직후 `/code-review`를 자동 실행, 인라인 코멘트까지 |
| 6. 머지 | ❌ 수동 (의도적) | 리뷰 확인 후 사용자가 "머지하자"고 하면 그때 `gh pr merge` |
| 7. 브랜치 정리 | ✅ `/pr` 8단계에 포함 | 머지 직후 `main`으로 이동 + pull + 로컬/원격 브랜치 삭제 |

**7단계가 필요한 이유**: 머지된 브랜치를 그대로 두고 다음 작업을 이어서 하면
(실제로 2026-07-12에 이 문제가 발생했다 — `feat/schedule-panel-day-colors`
브랜치 하나에서 PR #9와 #13이 연달아 머지됨), 이미 머지된 변경 이력과 다음
기능의 변경 이력이 한 브랜치에 계속 쌓인다. 매 기능은 `main`에서 새로 갈라져
나온 브랜치에서 시작해야 한다.

**머지를 의도적으로 자동화하지 않은 이유**: main에 반영되는 시점은 매번 사람이
확인하는 게 안전하다고 판단했다 (2026-07-12 합의). `/pr`은 PR 생성과 리뷰까지만
하고, 머지는 리뷰 결과를 보고 별도로 요청해야 진행된다.

## 1. 백로그 — `/issue`

"이슈로 추가해줘", "백로그에 넣어줘" 같은 요청에 반응한다.
`.github/ISSUE_TEMPLATE/feature_request.md` 구조를 그대로 따른다:

```
## 📋 개요
<왜 필요한지 — 지금 뭐가 불편한지, 무슨 요구사항에서 나왔는지>

## ✅ 작업 내용
1. <구체적 작업 단위 — 실제로 손댈 파일/기능 단위로 쪼갠다>

## 🎯 완료 조건
- [ ] <이 기능이 끝났다고 판단할 수 있는 구체적 기준>

## 🔗 참고
- <관련 파일 경로 / 관련 이슈·PR 번호>
```

- **제목/본문 모두 한글**로 작성한다.
- 라벨은 `enhancement` (버그면 `bug`).
- 여러 개를 한 번에 요청하면 각각 별도 이슈로 순서대로 만든다.

## 2. 브랜치 생성

이슈 하나를 골라 작업을 시작할 때, `docs/CONVENTIONS.md`의 브랜치 컨벤션
(`<type>/<짧은-이름>`)에 맞춰 새 브랜치를 만든다 — 짧은 이름은 이슈 제목(한글)의
뜻을 딴 영어 slug로 의역한다.

```bash
git checkout main && git pull origin main
git checkout -b feat/<짧은-이름>   # 버그 이슈면 fix/<짧은-이름>
```

## 3. PRD 업데이트

구현을 시작하기 전에 `docs/PRD.md`의 "기능별 요구사항"에 이 기능을 요구사항
항목으로 먼저 추가한다 — "무엇을, 왜" 만드는지 코드보다 먼저 문서로 확정해두는
것. (단, `docs/PROMPTS.md`처럼 문서 내용 자체가 곧 프롬프트 원문인 경우는 예외로
그 문서가 먼저다 — 일반적으로는 코드가 먼저고 레퍼런스 문서가 뒤따른다. 자세한
기준은 `docs/PROJECT.md` 대신 이 문서와 `CLAUDE.md`를 참고.)

## 4. 구현

이슈의 "작업 내용"/"완료 조건"을 기준으로 진행하고, 완료된 항목은 이슈에서
체크한다. 이 단계는 자동화 대상이 아니다 — 매 기능마다 코드가 다르므로 대화로
같이 짠다.

## 5. PR 생성 + 자동 리뷰 — `/pr`

"PR 올려줘" 요청에 반응한다.

1. `origin/main` 기준으로 커밋 범위를 확인하고 push
2. 제목 `Feat: <한글 설명>` / `Fix: <한글 설명>`, 본문은
   `.github/PULL_REQUEST_TEMPLATE.md` 구조(`## Summary` / `## Test plan`)
3. `gh pr create`로 PR 생성
4. 생성 직후 바로 `/code-review`를 실행해 diff를 리뷰하고, 발견한 이슈를 PR에
   인라인 코멘트로 남긴다 (`--comment` 옵션)

## 6. 확인 후 머지

`/pr`이 남긴 리뷰 코멘트를 사용자가 확인하고, 문제 없으면 머지를 요청한다.
그러면 `gh pr merge --merge`로 머지한다 (기존 PR들과 같은 방식 — squash나
rebase가 아니라 일반 merge commit).

머지 시 `main`과 브랜치 사이에 충돌이 있으면(예: 같은 문서를 서로 다른
브랜치에서 각자 업데이트한 경우) 어느 쪽 내용이 실제 코드 상태와 더 맞는지
확인해서 해결한 뒤 진행한다 — 무조건 최신 커밋을 우선하지 않는다.

## 7. 브랜치 정리

머지 직후 바로 이어서 진행한다:

```bash
git checkout main
git pull origin main
git branch -d <머지한-브랜치>
git push origin --delete <머지한-브랜치>
```

다음 기능은 이 정리된 `main`에서 다시 2단계(브랜치 생성)부터 시작한다 —
머지된 브랜치를 재사용해서 다음 작업을 이어가지 않는다.
