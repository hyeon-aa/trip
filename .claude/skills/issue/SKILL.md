# /issue — 기능 백로그 이슈 등록

`.github/ISSUE_TEMPLATE/feature_request.md` 구조(개요/작업 내용/완료 조건/참고)를
그대로 따른다. "기능 정하고 → PRD에 적고 → 구현 → `/pr`로 리뷰까지" 흐름의 첫 단계.

## 1. 아이디어 파악

대화에서 이미 논의된 기능이면 그 내용을 그대로 쓰고, 여러 개를 한 번에
요청하면 각각 별도 이슈로 순서대로 만든다. 애매하면 짧게 되물어서
"왜 필요한지"만 확정한다 — 나머지는 코드베이스를 보고 직접 추정해도 된다.

## 2. 초안 작성 (기존 이슈 톤 참고: `gh issue list --state all`)

```
## 📋 개요
<이 기능이 왜 필요한지 — 지금 뭐가 불편한지, 무슨 요구사항/이슈에서 나왔는지.
필요하면 근거(수치, 관련 코드 인용)를 짧게 곁들인다.>

## ✅ 작업 내용
1. <구체적 작업 단위 — 실제로 손댈 파일/기능 단위로 쪼갠다>
   - <세부 사항이나 결정해야 할 옵션이 있으면 하위 bullet으로>
2. <다음 작업 단위>

## 🎯 완료 조건
- [ ] <이 기능이 끝났다고 판단할 수 있는 구체적 기준>
- [ ] <PRD.md 업데이트 등 마무리 작업이 있으면 포함>

## 🔗 참고
- <관련 파일 경로>
- <관련 이슈/PR 번호>
```
**제목/본문 모두 한글로** 작성한다. 제목은 기존 이슈처럼 간결한 명사형
("일정에서 장소 삭제 버튼" 스타일).

## 3. 확인 후 생성

초안을 사용자에게 보여주고 확정되면(여러 개면 한 번에 다 보여주고 일괄 확정):
```bash
gh issue create --title "<한글 제목>" --label enhancement --body "$(cat <<'EOF'
<한글 본문>
EOF
)"
```
버그 리포트 성격이면 `enhancement` 대신 `bug` 라벨을 쓴다.

## 4. 작업 시작 (이슈 하나를 골랐을 때)

1. `docs/CONVENTIONS.md` 브랜치 컨벤션(`<type>/<짧은-이름>`)에 맞춰 이슈 제목에서
   뜻을 딴 짧은 영어 slug를 만들어 새 브랜치를 만든다 (제목 자체는 한글이므로
   영어로 의역):
   ```bash
   git checkout main && git pull origin main
   git checkout -b feat/<짧은-이름>
   ```
   (버그 이슈면 `fix/<짧은-이름>`.)
2. `docs/PRD.md` "기능별 요구사항"에 이슈 내용을 요구사항 항목으로 먼저 추가한다
   (기존에 합의한 "PRD 먼저" 흐름).
3. 이슈의 "작업 내용"/"완료 조건"을 기준으로 구현을 진행하고, 완료된 항목은
   이슈에서 체크한다.
4. 다 되면 `/pr`로 PR을 올린다.
