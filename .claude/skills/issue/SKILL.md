# /issue — 기능 백로그 이슈 등록

`.github/ISSUE_TEMPLATE/feature_request.md` 구조(왜/무엇/스코프)를 그대로 따른다.
"기능 정하고 → PRD에 적고 → 구현 → `/pr`로 리뷰까지" 흐름의 첫 단계.

## 1. 아이디어 파악

대화에서 이미 논의된 기능이면 그 내용을 그대로 쓰고, 여러 개를 한 번에
요청하면 각각 별도 이슈로 순서대로 만든다. 애매하면 짧게 되물어서
"왜 필요한지"만 확정한다 — "무엇"과 "스코프"는 코드베이스를 보고 직접
추정해도 된다.

## 2. 초안 작성 (기존 이슈 톤 참고: `gh issue list --state all`)

```
## 왜
<이 기능이 필요한 이유 — 지금 뭐가 불편한지/무슨 요구사항에서 나왔는지>

## 무엇
<구체적으로 뭘 만드는지, 1~3문장>

## 스코프
- client: <해당 시 컴포넌트/파일>
- server: <해당 시 도메인/엔드포인트, 없으면 항목 생략>
```
제목은 기존 이슈처럼 간결한 명사형 한글 ("일정에서 장소 삭제 버튼" 스타일).

## 3. 확인 후 생성

초안을 사용자에게 보여주고 확정되면(여러 개면 한 번에 다 보여주고 일괄 확정):
```bash
gh issue create --title "<제목>" --label enhancement --body "$(cat <<'EOF'
<본문>
EOF
)"
```
버그 리포트 성격이면 `enhancement` 대신 `bug` 라벨을 쓴다.

## 4. 다음 단계 안내

이슈 생성 후, 이 중 하나를 골라 시작하고 싶은지 물어본다 — 고르면
`docs/PRD.md` "기능별 요구사항"에 항목을 먼저 추가하고 나서 구현으로
넘어간다 (기존에 합의한 흐름).
